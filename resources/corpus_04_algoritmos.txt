U: oye, ¿estás familiarizado con el patrón de diseño Singleton?
B: Sí, es uno de los patrones más conocidos. Básicamente asegura que una clase tenga una única instancia en toda la aplicación, y proporciona un punto de acceso global a ella.

U: ¿y cuándo se recomienda usarlo?
B: Se recomienda usarlo cuando necesitas controlar el acceso a un recurso compartido, como una conexión a base de datos, un sistema de logs, o una configuración global.

U: pero también tiene sus desventajas, ¿no?
B: Sí, puede hacer que el código sea más difícil de probar porque introduce dependencias ocultas. Además, si no se usa correctamente, puede llevar a problemas de acoplamiento.

U: ¿qué alternativas hay al Singleton?
B: Puedes usar inyección de dependencias para gestionar la vida útil de las instancias, o usar un patrón de fábrica para controlar la creación sin crear un punto de acceso global.

U: ¿has usado alguna vez el patrón Observer?
B: Sí, es muy útil cuando necesitas que múltiples objetos reaccionen a cambios en otro objeto, como en sistemas de eventos o notificaciones.

U: ¿se parece al patrón Pub/Sub?
B: Sí, son similares. En Observer, el sujeto conoce a sus observadores. En Pub/Sub, los publicadores y suscriptores están desacoplados mediante un intermediario.

U: ¿cuál recomendarías más?
B: Depende de la arquitectura. Para sistemas acoplados, Observer. Para sistemas distribuidos o con bajo acoplamiento, Pub/Sub es mejor.

U: ¿y qué opinas del patrón MVC?
B: Es uno de los patrones más populares en aplicaciones web. Separar la lógica de negocio, la interfaz de usuario y el controlador facilita el mantenimiento y las pruebas.

U: ¿se puede usar en aplicaciones de escritorio también?
B: Sí, claro. En WPF se usa MVVM, que es una variante de MVC adaptada a la vinculación de datos.

U: ¿cuál es la diferencia entre MVC y MVVM?
B: En MVVM, el ViewModel actúa como un puente entre la vista y el modelo, usando enlaces de datos bidireccionales. En MVC, el controlador maneja las interacciones del usuario.

U: ¿qué patrón recomiendas para una API REST?
B: Recomiendo usar el patrón Repositorio para la capa de datos, y un patrón de Servicio para la lógica de negocio. También es importante manejar las excepciones con un middleware.

U: ¿qué es un middleware en ASP.NET Core?
B: Es un componente que se ejecuta en el pipeline de solicitudes. Puede manejar autenticación, logging, compresión, y mucho más. Es una forma flexible de agregar funcionalidad.

U: ¿cómo manejas la autenticación en tus APIs?
B: Normalmente uso JWT (JSON Web Tokens). Son ligeros, escalables y fáciles de implementar. También se pueden combinar con refresh tokens para mayor seguridad.

U: ¿qué es un refresh token?
B: Es un token de larga duración que se usa para obtener nuevos access tokens sin que el usuario tenga que iniciar sesión de nuevo. Se almacena de forma segura en el cliente.

U: ¿has probado OAuth2?
B: Sí, es un protocolo estándar para autorización. Permite que aplicaciones de terceros accedan a recursos del usuario sin compartir sus credenciales.

U: ¿y OpenID Connect?
B: OpenID Connect es una capa sobre OAuth2 que añade autenticación. Permite verificar la identidad del usuario, no solo autorizar acceso a recursos.

U: ¿qué base de datos prefieres para proyectos grandes?
B: Depende de los requisitos. Para datos relacionales con muchas relaciones, PostgreSQL. Para escalabilidad horizontal y datos no estructurados, MongoDB.

U: ¿has usado Entity Framework Core?
B: Sí, es el ORM principal de .NET. Facilita el trabajo con bases de datos, pero hay que tener cuidado con las consultas ineficientes.

U: ¿cómo evitas consultas lentas con Entity Framework?
B: Usando `AsNoTracking()` para consultas de solo lectura, cargando solo los datos necesarios con `Select()` y usando índices adecuados en la base de datos.

U: ¿qué opinas de Dapper como alternativa a Entity Framework?
B: Dapper es más ligero y rápido porque no tiene el overhead de EF. Es ideal cuando necesitas control total sobre las consultas SQL.

U: ¿cuándo usarías Dapper en lugar de EF?
B: Usaría Dapper en proyectos donde el rendimiento es crítico o cuando necesito ejecutar consultas complejas que EF no optimiza bien.

U: ¿qué es el patrón CQRS?
B: CQRS separa las operaciones de lectura y escritura en modelos diferentes. Permite optimizar cada lado de forma independiente.

U: ¿y el Event Sourcing?
B: Event Sourcing almacena el estado de una aplicación como una secuencia de eventos inmutables. Es útil cuando necesitas auditar cambios o reconstruir estados pasados.

U: ¿has combinado CQRS con Event Sourcing?
B: Sí, es una combinación poderosa para sistemas complejos, pero añade mucha complejidad. Hay que evaluar si realmente es necesario.

U: ¿qué es el patrón Unit of Work?
B: Es un patrón que agrupa múltiples operaciones de base de datos en una sola transacción. Asegura que todas se completen o ninguna, manteniendo la consistencia.

U: ¿has usado alguna vez el patrón Repository?
B: Sí, es muy común para abstraer el acceso a datos. Facilita las pruebas unitarias y el cambio de la fuente de datos.

U: ¿qué ventajas tiene usar el patrón Repository?
B: Desacopla la lógica de negocio de la lógica de acceso a datos. Hace el código más mantenible y fácil de probar.

U: ¿y qué desventajas tiene?
B: Puede añadir una capa extra de abstracción que no siempre es necesaria. Además, si no se usa correctamente, puede causar problemas de rendimiento.

U: ¿cómo manejas el versionado de una API?
B: Puedes hacerlo mediante la URL, como `/api/v1/usuarios`, o mediante headers. Cada versión puede tener cambios sin romper la compatibilidad.

U: ¿qué estrategia usas para versionar?
B: Prefiero el versionado por URL porque es más explícito y fácil de documentar. También permite mantener múltiples versiones activas simultáneamente.

U: ¿qué es una API RESTful?
B: Es una API que sigue los principios de REST: recursos identificados por URLs, métodos HTTP (GET, POST, PUT, DELETE), y representaciones como JSON o XML.

U: ¿qué es GraphQL?
B: GraphQL es un lenguaje de consulta para APIs que permite al cliente solicitar exactamente los datos que necesita, evitando la sobrecarga o subcarga de datos.

U: ¿qué ventajas tiene GraphQL sobre REST?
B: Reduce el número de peticiones, permite consultas más flexibles y evita el versionado con cambios incrementales.

U: ¿y qué desventajas tiene GraphQL?
B: Puede ser más complejo de implementar y puede haber problemas de rendimiento si no se optimizan las consultas.

U: ¿has usado alguna vez gRPC?
B: Sí, gRPC es un framework de RPC de alto rendimiento basado en HTTP/2 y Protocol Buffers. Es ideal para microservicios.

U: ¿qué ventajas tiene gRPC?
B: Es muy rápido, tiene soporte para streaming bidireccional, y es multiplataforma. Además, los mensajes binarios son más ligeros que JSON.

U: ¿cuándo usarías gRPC en lugar de REST?
B: Usaría gRPC para comunicaciones internas entre microservicios donde el rendimiento es clave. Para APIs públicas, seguiría usando REST.

U: ¿qué es Docker?
B: Docker es una plataforma que permite empaquetar aplicaciones en contenedores, asegurando que funcionen igual en cualquier entorno.

U: ¿y Kubernetes?
B: Kubernetes es un orquestador de contenedores que automatiza el despliegue, escalado y gestión de aplicaciones en contenedores.

U: ¿has desplegado aplicaciones en Kubernetes?
B: Sí, he trabajado con Kubernetes en varios proyectos. Es una herramienta muy potente pero requiere una curva de aprendizaje.

U: ¿qué es un Helm chart?
B: Es un paquete que define, instala y actualiza aplicaciones en Kubernetes. Facilita el despliegue repetible.

U: ¿qué es CI/CD?
B: CI es Integración Continua, CD es Despliegue Continuo. Automatizan la construcción, prueba y despliegue de software.

U: ¿qué herramientas usas para CI/CD?
B: He usado GitHub Actions, GitLab CI y Azure DevOps. Cada una tiene sus ventajas, pero GitHub Actions es muy integrado con el ecosistema.

U: ¿qué es un pipeline de CI/CD?
B: Es una secuencia de pasos automatizados que construyen, prueban y despliegan el código. Puede incluir tests, escaneo de seguridad y notificaciones.

U: ¿cómo aseguras la calidad del código en un pipeline?
B: Añadiendo pruebas unitarias, pruebas de integración, análisis de cobertura de código y linters como SonarQube.

U: ¿qué es SonarQube?
B: Es una herramienta de análisis de código que detecta bugs, vulnerabilidades y problemas de mantenibilidad.

U: ¿qué es el análisis estático de código?
B: Es el análisis del código fuente sin ejecutarlo. Busca problemas como código muerto, vulnerabilidades o malas prácticas.

U: ¿qué es el análisis dinámico?
B: Es el análisis del código en ejecución. Busca problemas de memoria, fugas, o comportamientos inesperados.

U: ¿has trabajado con microservicios?
B: Sí, he diseñado y desarrollado microservicios. Es una arquitectura que favorece la escalabilidad y el desacoplamiento.

U: ¿qué desafíos has encontrado con microservicios?
B: La comunicación entre servicios, la gestión de transacciones distribuidas y el monitoreo son los principales desafíos.

U: ¿cómo resuelves la comunicación entre microservicios?
B: Usando mensajería asíncrona con RabbitMQ o Kafka, o comunicación síncrona con gRPC o REST.

U: ¿qué es un message broker?
B: Es un intermediario que gestiona el intercambio de mensajes entre servicios. Permite comunicación desacoplada y escalable.

U: ¿qué es Kafka?
B: Kafka es una plataforma de streaming distribuida. Es ideal para sistemas de eventos en tiempo real y procesamiento de datos a gran escala.

U: ¿y RabbitMQ?
B: RabbitMQ es un message broker basado en AMQP. Es más ligero y fácil de configurar que Kafka.

U: ¿cuál elegirías?
B: Depende de las necesidades. Kafka para streaming de alto rendimiento y RabbitMQ para colas de tareas y comunicación simple.

U: ¿qué es una arquitectura hexagonal?
B: Es una arquitectura que aísla la lógica de negocio de los adaptadores externos. Facilita el testing y el mantenimiento.

U: ¿qué es una arquitectura limpia?
B: Es similar a la hexagonal, pero con capas más definidas: entidades, casos de uso, interfaces y frameworks.

U: ¿cuál prefieres?
B: Ambas son buenas. La arquitectura limpia es más estructurada, mientras que la hexagonal es más flexible.

U: ¿qué es DDD?
B: DDD (Domain-Driven Design) es una metodología que se centra en modelar el dominio del negocio con un lenguaje común entre expertos y desarrolladores.

U: ¿qué es un agregado en DDD?
B: Es un clúster de objetos que se trata como una unidad. Tiene una raíz de agregado que controla el acceso a los objetos internos.

U: ¿qué es un evento de dominio?
B: Es un evento significativo en el dominio que otros componentes pueden escuchar. Ayuda a desacoplar componentes.

U: ¿qué es un value object?
B: Es un objeto inmutable que se define por sus atributos. No tiene identidad propia, como una dirección o una cantidad de dinero.

U: ¿qué diferencia hay entre un value object y una entidad?
B: Una entidad tiene identidad propia (un ID), mientras que un value object se identifica por sus atributos.

U: ¿qué es un repositorio en DDD?
B: Es un patrón que proporciona una abstracción para el acceso a los agregados. Oculta los detalles de persistencia.

U: ¿qué es una fábrica en DDD?
B: Es un patrón que encapsula la creación de objetos complejos. Separa la construcción de la lógica de negocio.

U: ¿qué es un servicio de dominio?
B: Es un servicio que contiene lógica de negocio que no pertenece a una entidad o value object. Suele operar sobre múltiples objetos.

U: ¿qué es un servicio de aplicación?
B: Es un servicio que orquesta casos de uso. Coordina la lógica de negocio y la infraestructura.

U: ¿cómo pruebas un servicio de aplicación?
B: Usando mocks para las dependencias de infraestructura. Así pruebas la lógica de orquestación de forma aislada.

U: ¿qué es un mock?
B: Es un objeto falso que simula el comportamiento de una dependencia real. Se usa en pruebas unitarias.

U: ¿qué es un stub?
B: Es similar a un mock, pero solo devuelve respuestas predefinidas. No verifica cómo se llama.

U: ¿qué es un spy?
B: Es un objeto que registra cómo se usa, permitiendo verificar que se llamó correctamente.

U: ¿qué es TDD?
B: TDD (Test-Driven Development) es una metodología donde se escriben las pruebas antes del código.

U: ¿cuáles son los pasos de TDD?
B: Rojo (escribe la prueba que falla), Verde (escribe el código mínimo para que pase), Refactor (mejora el código sin cambiar su comportamiento).

U: ¿has aplicado TDD en proyectos reales?
B: Sí, en algunos proyectos. Es muy efectivo, pero requiere disciplina y puede ralentizar el desarrollo inicial.

U: ¿qué es BDD?
B: BDD (Behavior-Driven Development) es una extensión de TDD que se centra en el comportamiento del sistema desde la perspectiva del usuario.

U: ¿qué lenguaje se usa en BDD?
B: Gherkin, que permite escribir escenarios en lenguaje natural con formato Given/When/Then.

U: ¿qué es Given/When/Then?
B: Es un formato para escribir escenarios de prueba: Given (dado un contexto), When (cuando ocurre una acción), Then (entonces se espera un resultado).

U: ¿qué herramienta usas para BDD?
B: SpecFlow para .NET, y Cucumber para otros lenguajes.

U: ¿qué es el desarrollo ágil?
B: Es un enfoque iterativo para el desarrollo de software, centrado en la colaboración, la flexibilidad y la entrega continua.

U: ¿qué es Scrum?
B: Scrum es un marco ágil con roles como Product Owner, Scrum Master y Development Team. Incluye sprints, backlog y retrospectivas.

U: ¿qué es un sprint?
B: Es un período de tiempo fijo (normalmente 1-4 semanas) en el que se completa un conjunto de tareas del backlog.

U: ¿qué es Kanban?
B: Kanban es un método ágil que visualiza el trabajo en un tablero, limitando el trabajo en progreso (WIP).

U: ¿cuál prefieres, Scrum o Kanban?
B: Depende del proyecto. Scrum para proyectos con entregas definidas, Kanban para equipos de soporte o flujo continuo.

U: ¿qué es el pair programming?
B: Es una práctica donde dos programadores trabajan juntos en el mismo código. Uno escribe, el otro revisa en tiempo real.

U: ¿qué ventajas tiene?
B: Mejora la calidad del código, facilita el intercambio de conocimiento y reduce los errores.

U: ¿has hecho pair programming?
B: Sí, y aunque al principio puede ser incómodo, es muy efectivo a largo plazo.

U: ¿qué es code review?
B: Es la práctica de revisar el código de otros desarrolladores antes de fusionarlo al repositorio principal.

U: ¿qué se busca en un code review?
B: Se busca detectar errores, mejorar la legibilidad, asegurar que sigue las guías de estilo y que está bien testeado.

U: ¿qué herramientas usas para code review?
B: GitHub Pull Requests, GitLab Merge Requests y Azure DevOps Pull Requests.

U: ¿qué es una pull request?
B: Es una solicitud para fusionar los cambios de una rama con otra. Incluye comentarios y revisión de código.

U: ¿qué es un merge conflict?
B: Es un conflicto que ocurre cuando dos ramas tienen cambios incompatibles en el mismo archivo.

U: ¿cómo resuelves un merge conflict?
B: Manualmente, eligiendo qué cambios conservar, o usando herramientas de resolución de conflictos.

U: ¿qué es Git?
B: Es un sistema de control de versiones distribuido. Permite rastrear cambios en el código y colaborar con otros.

U: ¿cuáles son los comandos básicos de Git?
B: git init, git add, git commit, git push, git pull, git branch, git merge, git clone.

U: ¿qué es git stash?
B: Guarda los cambios no confirmados temporalmente para limpiar el directorio de trabajo sin perderlos.

U: ¿qué es git rebase?
B: Rebase reescribe el historial de commits, moviendo los cambios de una rama a otra de forma lineal.

U: ¿cuándo usarías rebase en lugar de merge?
B: Cuando quieres mantener un historial lineal y limpio. Pero hay que tener cuidado porque reescribe el historial.

U: ¿qué es git cherry-pick?
B: Aplica un commit específico de una rama a otra, sin fusionar toda la rama.

U: ¿qué es GitFlow?
B: Es un modelo de branching con ramas como develop, feature, release y hotfix. Organiza el flujo de trabajo en equipos grandes.

U: ¿qué es GitHub Flow?
B: Es un modelo más simple, con una rama principal (main) y ramas de características que se fusionan mediante pull requests.

U: ¿cuál prefieres?
B: GitHub Flow para equipos pequeños y despliegues continuos. GitFlow para proyectos con versiones programadas.

U: ¿qué es la integración continua?
B: CI es la práctica de fusionar cambios frecuentemente en una rama compartida, con pruebas automatizadas para detectar errores.

U: ¿qué es el despliegue continuo?
B: CD es la práctica de desplegar automáticamente los cambios que pasan las pruebas a producción.

U: ¿qué es el despliegue continuo y la entrega continua?
B: Entrega continua (CD) es tener el código listo para desplegar. Despliegue continuo es desplegar automáticamente cada cambio.

U: ¿qué es un feature flag?
B: Es una técnica que permite activar o desactivar funcionalidades en producción sin necesidad de hacer un nuevo despliegue.

U: ¿qué ventajas tiene el uso de feature flags?
B: Permite pruebas A/B, despliegues más seguros y la capacidad de desactivar funciones problemáticas al instante.

U: ¿qué es el canary deployment?
B: Es una estrategia de despliegue donde se lanza una nueva versión a un pequeño grupo de usuarios antes de hacerlo a todos.

U: ¿qué es blue-green deployment?
B: Es una estrategia donde se mantienen dos entornos idénticos (blue y green). Uno está activo y el otro recibe la nueva versión.

U: ¿qué es el rollback?
B: Es el proceso de volver a una versión anterior de la aplicación en caso de que la nueva tenga problemas.

U: ¿cómo haces rollback en producción?
B: Con pipelines automatizados, usando versionado de imágenes de contenedores o mediante feature flags.

U: ¿qué es un health check?
B: Es un endpoint que verifica el estado de una aplicación. Se usa para determinar si el servicio está listo para recibir tráfico.

U: ¿qué es un readiness probe?
B: Es una sonda que verifica si una aplicación está lista para recibir tráfico en Kubernetes.

U: ¿qué es un liveness probe?
B: Es una sonda que verifica si una aplicación sigue funcionando. Si falla, Kubernetes reinicia el contenedor.

U: ¿qué es el observability?
B: Es la capacidad de entender el estado interno de un sistema a través de logs, métricas y trazas.

U: ¿qué son los logs?
B: Son registros de eventos generados por la aplicación. Ayudan a diagnosticar problemas y entender el comportamiento.

U: ¿qué son las métricas?
B: Son datos cuantitativos sobre el rendimiento del sistema, como tiempo de respuesta, uso de CPU o memoria.

U: ¿qué son las trazas?
B: Son registros de la ruta de una solicitud a través de un sistema distribuido. Permiten ver el flujo completo de una petición.

U: ¿qué herramienta usas para logging?
B: Serilog y Application Insights. Serilog es muy flexible y fácil de configurar.

U: ¿qué es el structured logging?
B: Es un enfoque de logging donde los eventos se registran con formato estructurado (ej. JSON), facilitando su búsqueda y análisis.

U: ¿qué es un telemetry pipeline?
B: Es un sistema que recoge, procesa y almacena datos de telemetría para análisis y monitoreo.

U: ¿qué es un dashboard en observability?
B: Es una interfaz gráfica que muestra métricas clave en tiempo real, como el estado de los servicios y el rendimiento.

U: ¿qué es Grafana?
B: Es una herramienta de visualización que permite crear dashboards interactivos a partir de datos de monitoreo.

U: ¿qué es Prometheus?
B: Es un sistema de monitoreo y alerta de código abierto, con un modelo de datos de series temporales.

U: ¿qué es ELK Stack?
B: Es un conjunto de herramientas: Elasticsearch (búsqueda), Logstash (procesamiento) y Kibana (visualización).

U: ¿qué es un APM?
B: APM (Application Performance Monitoring) es una herramienta que monitoriza el rendimiento de aplicaciones en tiempo real.

U: ¿qué es el logging distribuido?
B: Es la práctica de centralizar logs de múltiples servicios para facilitar la correlación y el análisis.

U: ¿qué es el correlation ID?
B: Es un identificador único que se pasa a través de las llamadas entre servicios, permitiendo rastrear una solicitud completa.

U: ¿cómo implementas un correlation ID?
B: Se genera al inicio de la solicitud y se añade a los headers de todas las llamadas posteriores.

U: ¿qué es un gateway de API?
B: Es un punto de entrada único para las solicitudes a microservicios. Gestiona el enrutamiento, autenticación y limitación de velocidad.

U: ¿qué es un rate limiter?
B: Es un mecanismo que limita el número de solicitudes que un usuario puede hacer en un período de tiempo.

U: ¿qué es un circuit breaker?
B: Es un patrón que previene que un servicio siga llamando a otro que está fallando, permitiendo una recuperación más rápida.

U: ¿cómo implementas un circuit breaker?
B: Con librerías como Polly en .NET. Configuras umbrales de fallos y tiempos de espera antes de reintentar.

U: ¿qué es una retry policy?
B: Es una política que define cuántas veces y con qué intervalos se debe reintentar una operación fallida.

U: ¿qué es el jitter en las retries?
B: Es un retraso aleatorio entre reintentos para evitar el efecto "thundering herd" cuando muchos servicios reintentan al mismo tiempo.

U: ¿qué es el bulkhead pattern?
B: Es un patrón que aísla recursos para evitar que un fallo en un servicio afecte a otros. Similar a los compartimentos de un barco.

U: ¿qué es la estrategia de backoff exponencial?
B: Es una estrategia donde el tiempo de espera entre reintentos se duplica progresivamente.

U: ¿qué es un timeout?
B: Es el tiempo máximo que se espera por una respuesta antes de cancelar la operación.

U: ¿cómo configuras timeouts en aplicaciones web?
B: En el cliente HTTP, en el middleware y en las bases de datos. Es importante tenerlos en todos los niveles.

U: ¿qué es el health check en una API?
B: Es un endpoint que devuelve el estado de salud del servicio (ej. /health). Es usado por sistemas de orquestación.

U: ¿qué es la tolerancia a fallos?
B: Es la capacidad de un sistema para seguir funcionando correctamente a pesar de fallos en algunos de sus componentes.

U: ¿qué es el diseño para fallos?
B: Es el enfoque de construir sistemas asumiendo que los fallos van a ocurrir y diseñar para manejarlos.

U: ¿qué es la resiliencia?
B: Es la capacidad de un sistema para recuperarse rápidamente de fallos y seguir proporcionando el servicio.

U: ¿qué es la disponibilidad?
B: Es el porcentaje de tiempo que un sistema está operativo y accesible.

U: ¿qué es un SLA?
B: Service Level Agreement — es un acuerdo formal que define el nivel de servicio esperado, incluyendo disponibilidad y rendimiento.

U: ¿qué es un SLO?
B: Service Level Objective — es un objetivo específico dentro de un SLA, como "el 99.9% de las solicitudes deben responder en menos de 200ms".

U: ¿qué es un SLI?
B: Service Level Indicator — es una métrica que mide el cumplimiento del SLO, como "el tiempo medio de respuesta".

U: ¿qué es el error budget?
B: Es la cantidad de fallos permitidos dentro de un SLO. Si se excede, se deben priorizar mejoras de estabilidad.

U: ¿qué es el toil en SRE?
B: Es el trabajo manual y repetitivo que puede automatizarse. Los SRE intentan minimizar el toil.

U: ¿qué es SRE?
B: Site Reliability Engineering — es una disciplina que aplica principios de ingeniería de software a la gestión de infraestructura.

U: ¿qué es el chaos engineering?
B: Es la práctica de introducir fallos controlados en un sistema para probar su resiliencia.

U: ¿has practicado chaos engineering?
B: En entornos controlados, sí. Herramientas como Chaos Monkey ayudan a probar la robustez del sistema.

U: ¿qué es una arquitectura orientada a eventos?
B: Es una arquitectura donde los componentes se comunican mediante eventos, permitiendo un alto grado de desacoplamiento.

U: ¿qué es una arquitectura basada en mensajes?
B: Es similar a la arquitectura orientada a eventos, pero se enfoca en el intercambio de mensajes entre servicios.

U: ¿qué es una arquitectura sin servidor (serverless)?
B: Es un modelo donde el proveedor de la nube gestiona la infraestructura y los recursos se asignan bajo demanda.
